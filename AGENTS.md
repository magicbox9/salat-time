# Development instructions — Salat Time

This document is the source of truth for how work on this project is done.
Read it before starting any task. It applies to humans and AI assistants alike.

---

## 1. Architecture in one paragraph

Single-file Swift app (`main.swift`) compiled directly with `swiftc` from
`build.sh`. No Xcode project, no Swift Package Manager, no CocoaPods.
AppKit + WebKit + AVFoundation + CoreLocation + UserNotifications.
Target macOS 14+. All user prefs live in `UserDefaults` under `com.salattime.app`.
At build time `build.sh` downloads adhan + adhkar audio and the Cairo font.

The app has **two UI surfaces** (v3):
- **Menu bar** (always): status icon with countdown, left-click popover for
  quick prayer-time view, right-click menu for actions.
- **Main window** (on demand): a full management window opened from the menu
  bar's "Open Main Window…" item or the dock. Hosts tabs for Prayer Times,
  Adhkar, Adhan, Settings. Switches the app to `.regular` activation policy
  (dock icon appears) while open; back to `.accessory` (tray-only) on close.

## 4. Versioning (Semantic Versioning)

The app version follows **Semantic Versioning**: `MAJOR.MINOR.PATCH`

- **MAJOR** — incompatible / breaking changes (new app architecture, dropped
  macOS support, changed bundle ID, rebrand).
- **MINOR** — new features, backwards-compatible (added adhkar, added a window,
  new settings tab).
- **PATCH** — bug fixes, internal refactors, asset updates, no new behavior.

The version lives in **three places that MUST stay in sync**:

1. `main.swift` → `let kAppVersion = "x.y.z"`
2. `Info.plist` → `CFBundleVersion`
3. `Info.plist` → `CFBundleShortVersionString`

Bump the version as part of the same commit that introduces the change.
Rule of thumb:
- Each new feature → bump MINOR (2.2.0 → 2.3.0)
- Each bug fix → bump PATCH (2.2.0 → 2.2.1)
- Architectural overhaul → bump MAJOR (2.x → 3.0.0)

## 5. Commit system (Conventional Commits)

Every commit message follows the **Conventional Commits** spec:

```
<type>(<scope>): <short imperative summary in lowercase>

<optional body — what + why, wrap at ~72 chars>

<optional footer>
```

### Types (use only these)

| type     | when to use                                                       |
|----------|-------------------------------------------------------------------|
| `feat`   | a new feature visible to the user (new adhkar set, new window)    |
| `fix`    | a bug fix (decoding crash, layout bug, wrong time)                |
| `refactor` | internal restructure, no behavior change                        |
| `perf`   | performance improvement                                            |
| `docs`   | documentation only (README, AGENTS.md, HANDOFF.md)                |
| `build`  | changes to build.sh, Info.plist, signing, asset pipeline           |
| `style`  | formatting, whitespace, naming — no code logic                    |
| `chore`  | maintenance, gitignore, tooling                                    |
| `revert` | reverting a previous commit                                        |

### Scope (optional but encouraged)

The area of the codebase. Examples: `adhkar`, `adhan`, `ui`, `settings`,
`i18n`, `build`, `assets`, `menu`, `notifications`.

### Rules

1. **Subject line ≤ 72 chars**, lowercase, imperative mood
   ("add adhkar editor" ✅, not "Added Adhkar Editor" ❌).
2. **Subject stands alone** — no period at the end.
3. **Body explains the *why*, not the *what*** — the diff already shows what.
4. **One logical change per commit** — don't mix a feature with a refactor.
5. **Reference the issue** if applicable: `closes #12`, `refs #15`.
6. **Bump version in the same commit** as the change it corresponds to.

### Examples

```
feat(adhkar): add morning/evening adhkar with synced recitation

Adds a floating NSPanel that recites the 25 morning / 23 evening adhkar
from Hisn al-Muslim. Each dhikr's audio plays in order and the matching
Arabic text is shown — per-dhikr sync. Auto-triggers at sunrise (morning)
and Asr (evening); also accessible from the right-click menu.
Bumps version to 2.2.0.

fix(adhkar): handle _meta key in adhkar.json decoding

The decoder expected every top-level key to be an adhkar array; the
_meta metadata dict failed the type check and silently emptied the
session. Parse loosely now, only decoding known set keys.
Bumps version to 2.2.1.
```

## 6. Commit cadence

**Commit after every major milestone, not at the end.** This keeps the
history reviewable and gives safe rollback points. Specifically:

- After each phase of a multi-step feature (data, model, UI, wiring).
- After each bug fix (separate commit from the feature).
- Before starting a risky refactor (so it's easy to revert).
- Before any operation that touches the deployed app.

If a feature spans many files, prefer several small commits over one big one.

## 7. Before committing — checklist

1. `./update.sh --build` compiles cleanly with no errors.
2. Version is bumped in all three places (main.swift + Info.plist ×2).
3. `git status` shows only the intended changes.
4. Commit message follows the format in section 5.
5. The commit does not include `update.sh` or `Update.command`
   (gitignored — machine-specific).

## 8. Never commit

- `update.sh`, `Update.command` (machine-specific deploy helpers).
- `Salat Time.app/` (build artifact).
- Downloaded audio / fonts (regenerated by build.sh).
- `.DS_Store`.

These are already in `.gitignore`.

## 9. Adhkar collections (v3) — data model + storage

The adhkar feature (v3) uses a user-editable Collection model:

- **`AdhkarCollection`** — a named, ordered group of adhkar items with an
  `anchorKind` (manual / shuruq / fajr / asr / maghrib / isha) and an
  `autoPlay` flag. Stored as JSON in `UserDefaults` under `kAdhkarLibraryKey`.
- **`AdhkarEntry`** — one dhikr inside a collection. `audioRef` is a logical
  ref (`"bundled:75.mp3"` or `"imported:abc.mp3"`), resolved at runtime via
  `resolveAdhkarAudio()` — never store absolute paths.
- **Bundled library** — `adhkar.json` (read-only starter library, sourced
  from Hisn al-Muslim) + 27 MP3s in `Resources/adhkar_audio/`. The user pulls
  from these via "Add from Library…". Never overwritten.
- **Imported audio** — copied to `~/Library/Application Support/SalatTime/audio/`
  on import via `importAdhkarAudio(from:)`. NOT in the app bundle (the bundle
  is wiped on each `./update.sh` rebuild).
- **Migration** — `migrateAdhkarDefaults()` runs once on first v3 launch,
  seeding Morning (sunrise) + Evening (asr) collections from the bundled
  library so existing v2 behavior is preserved. Idempotent via
  `kAdhkarMigrated`.

The main window's Adhkar tab hosts the editor: two-pane (collections list
on the left, items on the right). Schedule (anchor + autoplay) is set via
each collection row's right-click menu.
